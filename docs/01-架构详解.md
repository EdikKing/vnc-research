## 每层做什么

### 1. Xvfb (X Virtual Framebuffer)

**是什么**: 无头 X server,在内存里虚拟出一块屏幕。

**为什么需要**: 大多数 GUI 程序(包括 chromium)都需要 X server 才能运行。物理服务器通常没有显示器,Xvfb 解决了"没屏幕但要跑 GUI"的问题。

**关键参数**:
- `:99` 显示编号(0 通常是物理显示,99 是我们常用的虚拟显示)
- `-screen 0 1280x800x24` 1280×800 像素,24 位色
- `-ac` 允许匿名连接(避免 Xauthority 麻烦)
- `-noreset` 不因为最后一个 client 退出就关掉

### 2. x11vnc

**是什么**: 把 X11 屏幕变成 VNC 协议的服务。

**为什么需要**: VNC 是跨平台远程桌面协议,几乎所有设备都能看(Web 端、移动端、PC 端都有 VNC 客户端)。

**关键参数**:
- `-display :99` 共享这个 X 显示
- `-forever` client 断开不退出
- `-shared` 多人同时连(必要 — agent + 你同时看)
- `-rfbport 5900` VNC 标准端口(v0.3.0 起无鉴权,只依赖 ngrok basic-auth 防外网访问)

### 3. websockify + noVNC

**是什么**: noVNC 是用 HTML5 Canvas 实现的 VNC 客户端,跑在浏览器里;但浏览器只能用 WebSocket,所以中间需要 websockify 把 WebSocket 转换成 VNC 原始 TCP 协议。

**为什么需要**: 你不需要装 VNC 客户端,直接浏览器打开网页就能看远程桌面。

**关键参数**:
- `--web=/usr/share/novnc` noVNC 的 HTML/JS 文件位置
- `6080 localhost:5900` 把 6080 端口的 WebSocket 请求代理到 5900 端口的 VNC

### 4. chromium (with CDP)

**是什么**: 浏览器。带 `--remote-debugging-port=9222` 后,会启动一个 CDP (Chrome DevTools Protocol) 服务。

**为什么需要**:
- 浏览器能渲染 JS SPA(知乎/小红书/抖音都是 SPA)
- `--remote-debugging-port` 让外部程序能通过标准协议控制它
- 跟 Playwright/Puppeteer 完美配合

**CDP 是什么**: Chrome DevTools Protocol,Google 官方协议,功能包括:导航、点击、输入、JS 执行、网络拦截、cookie 管理、性能分析等。

### 5. playwright

**是什么**: Microsoft 出的浏览器自动化库(底层用 CDP),API 比 Puppeteer 更友好,跨浏览器。

**为什么用 Playwright 而不是 Puppeteer**:
- API 更 Pythonic
- 自动等待(智能 retry)
- 跨浏览器(Firefox/WebKit/Chromium)
- 文档清晰

## 数据流(一次典型调研任务)

```python
用户: "调研小红书 AI 编程"
   │
   ▼
AI Agent 加载 skills/SKILL.md
   │
   ▼
playwright connect_cdp(9222) → 拿到 browser 对象
   │
   ▼
page.goto("https://xiaohongshu.com/search_result?keyword=AI编程")
   │
   ▼
page.wait_for_timeout(3000) # 等异步加载
   │
   ▼
page.evaluate("() => 抓所有 section.note-item 的数据") → 22 条搜索结果
   │
   ▼
对每条结果:page.goto(详情页 URL) → 抓标题/作者/正文 → 20 条完整数据
   │
   ▼
排序 + 输出 Markdown 报告给用户
   │
   ▼
[如果遇到登录墙]
Agent: "Operator, please login in VNC, then say 'done'"
   │
   ▼
Agent: 暂停,等你操作
   │
   ▼
你: 在 VNC 浏览器手动登录(cookie 写入 chromium profile)
   │
   ▼
你: "搞定了"
   │
   ▼
Agent: page.reload() → 继续抓数据
```

## 为什么这套设计 work

### 1. 完全内网
- noVNC 监听 0.0.0.0:6080,只暴露在你家/公司内网
- chromium DevTools 监听 127.0.0.1:9222,只接受本机连接
- **不需要公网 IP / ngrok / cloudflared**
- **不存在数据过第三方服务器**

### 2. 登录态复用
- chromium profile 在 `$HOME/.config/chromium`,包含所有 cookie/localStorage
- 你在 VNC 里登录一次小红书 → agent 用同一 chromium 自动登录态
- **不像 BrowserAct 每次都要重新登录**

### 3. 0 成本
- 所有组件都是开源免费
- 不消耗任何 AI token / credit / 配额
- agent 操作的"基础设施"免费

### 4. 人类 AI 共享
- 你和 agent 看的是同一块屏幕
- 你可以**实时看到** agent 在做什么
- agent 卡住时,你随时介入(在 VNC 手动操作)
- 介入完成后,cookie 自动写入,agent 继续

### 5. 可扩展
- 同样的架构可以扩展到:
  - 微信(Web 版)
  - Steam / Epic(游戏客户端)
  - VSCode(远程 IDE)
  - 任意桌面 GUI 应用

## 局限

1. **一台机器 = 一个人**(多用户需要不同端口组)
2. **stealth 不强**(datacenter IP 仍然会被拦,适合"已登录账号 + 国内平台"场景)
3. **agent 操作时 chromium 可能被风控**(国内大平台对自动化有检测)
4. **依赖稳定的网络**(noVNC 通过 WebSocket,网络差会卡)
