# 使用指南 · AI Agent + 人类协作协议

> 这套架构最大的特点是**人类和 AI 共享同一个浏览器**。理解协作协议才能用好。

---

## 核心原则

### "先试读再叫人"原则(本框架反复强调)

> 默认 agent 自动操作;遇到登录墙/验证码时才暂停叫人。

具体:
1. agent 接到任务 → 用 playwright 操控 chromium
2. **能直接拿到数据** → 直接抓,出报告
3. **遇到登录墙/滑块/验证码** → **暂停**,告诉操作员 "需要 VNC 操作",等操作员 "搞定了"再继续

**不要**:
- ❌ 默认每次都叫人介入
- ❌ 找不到数据就放弃(试过路径要在报告里列清楚)

### 🔴 暂停通知协议(标准模板)

agent 暂停时,**必须**按这个格式输出(关键:带 VNC URL):

```
🛑 暂停: 需要操作员介入
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
任务: <当前正在做什么,例如 "登录小红书">
原因: <具体卡在哪,例如 "搜索结果页显示 '登录后查看'">

👉 VNC 浏览器: http://<服务器IP>:6080/vnc.html
   (打开 → 看 chromium 当前页面 → 手动操作)

操作步骤:
1. 在 VNC 浏览器完成 <具体操作>
2. 操作完成后说 "搞定了"

agent 接下来会:
- 暂停(不调 playwright)
- 等你说 "搞定了"
- 然后自动 reload + 继续
━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**完整规则见** `../skills/SKILL.md` 的 "🔴 操作员介入协议" 章节。

## 标准协作协议

### 场景 1:正常抓取(0 介入)

```
User: "调研小红书 AI 编程"
   ↓
Agent: skill_view('chinese-web-research') # 加载 skill
   ↓
Agent: playwright connect_cdp(9222)
   ↓
Agent: page.goto("https://xiaohongshu.com/search_result?keyword=AI编程")
   ↓
Agent: page.evaluate("抓数据")
   ↓
[如果成功] → 输出报告
[如果登录墙] → 进入场景 2
```

### 场景 2:遇到登录墙

```
Agent: 检测到 "登录" 字样 / 验证码 iframe
   ↓
Agent: 暂停所有 playwright 操作
   ↓
Agent 输出: "操作员, 小红书需要登录。请在 VNC 浏览器手动登录,登录完告诉我 '搞定了'"
   ↓
[等待操作员操作]
   ↓
操作员: 在 VNC 浏览器手动登录小红书(扫码/手机号)
   ↓
操作员: "搞定了"
   ↓
Agent: page.reload() → 重新抓数据
```

### 场景 3:滑块验证码

```
Agent: 检测到 rmc.bytedance.com/verifycenter iframe
   ↓
Agent: 暂停,告诉操作员 "需要滑过滑块"
   ↓
操作员: 在 VNC 浏览器手动滑块
   ↓
操作员: "搞定了"
   ↓
Agent: page.reload() → 重新抓
```

### 场景 4:被风控拦截(IP 检测等)

```
Agent: 抓不到内容 + 不是登录墙 + 没有验证码
   ↓
Agent: 检查是不是 datacenter IP 风控
   ↓
[如果是] → 在报告里诚实标注 "❌ 此平台 datacenter IP 拦截,VNC 架构无法解决"
   ↓
Agent: 建议换路径(改用其他平台的数据 + 交叉验证)
```

## Agent 的标准任务流程

```python
from playwright.sync_api import sync_playwright
import time, json

def run_research_task(platform, query):
    with sync_playwright() as p:
        browser = p.chromium.connect_over_cdp("http://127.0.0.1:9222")
        page = browser.contexts[0].pages[0]
        
        # 1. 导航
        search_url = build_search_url(platform, query)
        page.goto(search_url)
        page.wait_for_timeout(3000)
        
        # 2. 检查登录态
        content = page.content()
        if "登录" in content and "登录后" in content:  # 登录墙
            print(f"⚠️  {platform} 需要登录")
            print("操作员, 请在 VNC 浏览器手动登录,登录完告诉我 '搞定了'")
            input("按 Enter 继续...")
            page.reload()
            page.wait_for_timeout(3000)
        
        # 3. 检查验证码
        if "verify" in page.content().lower() or "rmc.bytedance" in page.content():
            print(f"⚠️  {platform} 触发滑块验证")
            print("操作员, 请在 VNC 滑过滑块,完事告诉我 '搞定了'")
            input("按 Enter 继续...")
            page.reload()
            page.wait_for_timeout(3000)
        
        # 4. 抓数据
        data = extract_data(page, platform)  # 平台特定选择器
        return data
```

## 平台特定的踩坑(已验证)

### 小红书

| 坑 | 解法 |
|----|------|
| 搜索结果 `section.note-item a` 链接不是真 URL | 用 `a[href*="search_result/"]` 筛选(避开 `/explore/`) |
| 详情页 `/explore/` 返回 404 | 用搜索结果里的 `/search_result/<hash>?xsec_token=...` 链接 |
| 标题可能在 `<a>` 不在 `.title` | `.querySelector('.title') \|\| a.innerText` 兜底 |
| 点赞格式多样(87 / 1.2万 / 152) | 统一 `parse_int()` 函数处理 |

### 抖音

| 坑 | 解法 |
|----|------|
| 搜索页 = 登录墙 | 操作员先在 VNC 登录 |
| 滑块 iframe `rmc.bytedance.com` | 操作员手动滑 |
| 搜索结果被滑块挡住后只剩 AI 摘要 | 滑过滑块后再抓 |

### 知乎

| 坑 | 解法 |
|----|------|
| `HeadlessChrome` UA 被识别 → 40362 | VNC chromium 不用 HeadlessChrome,正常 chrome UA,直接通 |
| 搜索页要先登录 | 操作员在 VNC 登录一次就永久 |

## Agent 注意事项(必读)

### DO

- ✅ 每次任务**先加载 skill**(`skills/SKILL.md`)
- ✅ **用真实路径**(示例代码里给的命令复制就能用)
- ✅ 抓数据前**先 `page.wait_for_timeout(2000-4000)`** 等异步加载
- ✅ 选择器**加兜底**(`.querySelector('.title') || a.innerText`)
- ✅ 报告里**标注每条数据的可信度**(🟢/🟡/🔵/🔴/⚪)
- ✅ **暂停前明确告诉操作员** "需要你做什么" + "做完说什么"
- ✅ 抓到的数据**本地落盘**(`/tmp/xhs_<query>_<date>.json`)

### DON'T

- ❌ **不要**用 `curl + grep` 当主路径(对所有 SPA 都失败)
- ❌ **不要**默认每次都操作员介入(操作员已经在 VNC 里登录过了,cookie 直接用)
- ❌ **不要**替操作员填密码或点 OAuth(账号安全)
- ❌ **不要**关 chromium(丢失 profile / 登录态)
- ❌ **不要**假设登录态存在 — 用 `page.content()` 先确认

## 紧急情况处理

### Chromium 崩溃

```bash
# 健康检查
./scripts/health-check.sh
# 如果 chromium 没在,重启
./scripts/start-all.sh
```

profile 会自动加载(只要 `--user-data-dir` 对)。

### cookie 失效

操作员在 VNC 里手动重新登录该平台一次,**不需要重做整套架构**。

### agent 跑飞(操作错乱)

```bash
# 1. 强制关掉 chromium
pkill -f '/usr/lib/chromium/chromium'

# 2. 重启
./scripts/start-all.sh
```

操作员在 VNC 里检查 chromium 是否正确启动了,然后告诉 agent 继续。

## 给操作员看的"什么时候介入"

**操作员介入的场景**:

| 场景 | 触发条件 | 你要做什么 |
|------|---------|----------|
| 登录 | agent 说"X 平台需要登录" | 在 VNC 浏览器手动登录,说"搞定了" |
| 滑块 | agent 说"X 触发滑块" | 在 VNC 浏览器滑滑块,说"搞定了" |
| 验证码 | agent 说"X 触发验证码" | 在 VNC 浏览器输验证码,说"搞定了" |
| 错位 | VNC 看到 agent 操作错位置 | 在 VNC 手动修正,然后让 agent 重试 |

**操作员不介入的场景**:

- B站搜索(直接通)
- 头条搜索(直接通)
- 已登录平台正常抓取
- agent 自己能搞定的数据抓取

## 调试技巧

### 看 VNC 当前画面

直接在浏览器打开 `http://服务器:6080/vnc.html`,你会看到:
- agent 在做什么(实时)
- chromium 标签页
- 登录态是否还在

### 用 DevTools 看网络/控制台

VNC 浏览器里按 F12 打开 DevTools,跟普通 Chrome 一样用。可以帮 agent 诊断问题。

### agent 自己诊断

```python
# 看 console errors
errors = page.evaluate("() => window.__errors || []")

# 看网络请求
page.on("request", lambda req: print(f"REQ: {req.url}"))
page.on("response", lambda res: print(f"RES: {res.url} {res.status}"))

# 看 cookie
cookies = browser.contexts[0].cookies()
for c in cookies:
    if 'xiaohongshu' in c['domain']:
        print(c)
```
